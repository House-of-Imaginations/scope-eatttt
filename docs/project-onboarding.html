Your first hour

Read just enough to orient, then run the narrowest path that proves the app works on your machine.

  1. 1
    Read the product flow in README.md. The P1 loop is host creates session -> member joins -> everyone swipes -> candidates promote -> host opens poll -> votes -> winner.
  2. 2
    Skim AGENTS.md before edits. It defines the architecture boundaries, PgBouncer/LISTEN gotchas, and the repo rule to commit per COMMIT.md.
  3. 3
    Open the frontend handoff. It explains the current backend assumptions, the live/SSE expectations, and the remaining frontend integration direction.
  4. 4
    Trace one command: session.startSwiping. Follow it from packages/contract/src/router.ts to apps/web/src/lib/server/orpc.ts, packages/core/src/domain/session.ts, the DB repo, outbox event, SSE reducer, and apps/web/src/routes/s/[id]/+page.svelte.
  5. 5
    Run the app with fake providers first. Do not wire Google Places or OCR until the fake path works; the current local defaults are built for offline development.

Mental model

The app is intentionally split: typed commands change state, events move that state to browsers, and background jobs do slow work.

StartHost creates a lunch_session with location, cuisines, and radius.
JoinGuests join by code through anonymous Better Auth sessions.
SwipeEach member gets a deck and accepts or rejects restaurants.
PromoteA restaurant becomes a poll candidate after two distinct accepts.
VoteHost opens the poll; members upvote or downvote candidates.
DecidePoll closes by timer or host action; highest net score wins.
Commands

oRPC changes state

Mutations and query-style reads go through packages/contract and the SvelteKit oRPC handler.

Events

SSE updates browsers

Do not stream over oRPC. Browsers subscribe to GET /sessions/:id/events and replay with Last-Event-ID.

Jobs

BullMQ handles slow work

Places fetching and poll closing run in apps/worker, backed by Redis and the outbox event path.

Run it locally

Start with fake providers. Add real vendor keys only after the fake end-to-end loop works.

pnpm install
cp .env.example .env
docker compose up -d
pnpm db:migrate
pnpm --filter worker dev
pnpm --filter web dev

Use the right database URLs

DATABASE_URL points at PgBouncer on localhost:6432 for normal app queries.

DATABASE_DIRECT_URL points at Postgres on localhost:5432 for migrations and relay LISTEN.

The pooled postgres.js client must keep prepare: false.

Rules that matter

These are the constraints most likely to prevent subtle local-to-production breakage.

Keep core clean

Domain code depends on ports, not Google, Redis, BullMQ, Drizzle, or Better Auth.

Use the transactional outbox

Commands write domain rows and outbox_event rows in one transaction. Do not dual-write directly to Redis from command logic.

Separate commands from streams

Use oRPC for commands and reads. Use SSE for live updates and reconnect replay.

Keep Places server-side

Browser code never calls Places. Search is field-masked, cached for 30 minutes, and routed through adapters/jobs.

Do not hand-edit auth tables

Better Auth owns user, session, account, and verification. App tables FK to user.id.

OCR needs human confirmation

Phase 2 receipt OCR can suggest bill data, but it must never auto-finalize a bill.
